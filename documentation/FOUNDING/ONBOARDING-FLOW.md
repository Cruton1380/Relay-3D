#### **Step 5: Guardian Recovery Setup**
*[Detailed in USER-GUIDES/GUARDIAN-RECOVERY-USER-GUIDE.md]*

Guardian Recovery is Relay's social recovery system that empowers trusted friends to help you regain access to your account if you lose your device or face other access issues. Unlike traditional password reset systems, Guardian Recovery distributes trust across your personal network using cryptographic threshold schemes.

**How Guardian Recovery Works:**

1. **Cryptographic Sharing**: Your account recovery key is split into multiple encrypted shares using Shamir's Secret Sharing
2. **Distributed Trust**: These shares are securely distributed to your chosen guardians
3. **Threshold Recovery**: Only a predetermined number of guardians (threshold) need to assist for recovery
4. **Zero Knowledge**: Guardians never see or possess your actual recovery key
5. **Multi-Factor Verification**: Recovery requires both guardian assistance and additional identity verification

**Technical Implementation:**

```javascript

function createGuardianShares(accountKey, guardians, threshold) {
  // Generate recovery seed from account key

  const shares = shamirSplit(recoverySeed, {
    shares: guardians.length,
    threshold: threshold,  // Minimum needed for recovery

  const encryptedShares = guardians.map((guardian, i) => {

      guardianId: guardian.id,
      encryptedShare: encryptWithPublicKey(guardian.publicKey, shares[i])

}
```

Proximity Verification is Relay's innovative solution to the Sybil attack problemâ€”ensuring that each account represents a unique human by requiring real-world physical presence during onboarding. **Why Proximity Matters:**

Physical proximity verification solves multiple critical problems in digital communities:

1. **Sybil Resistance**: Prevents one person from creating multiple accounts
2. **Human Verification**: Confirms that accounts represent actual humans

4. **Community Integration**: Creates meaningful social connections

### **Phase 7: Governance Preparation**

#### **Step 11: Civic Identity Verification**

Civic Identity Verification creates a privacy-preserving link between a user's Relay identity and their real-world geographic presence, enabling meaningful participation in local governance while preventing multiple voting accounts. This system uses zero-knowledge proofs to verify residency without storing personally identifiable information.

**Regional Governance Innovation:**

Relay's approach to local governance solves critical challenges in digital democracy:

1. **Local Relevance**: Ensures decisions are made by actual community members
2. **Sybil Resistance**: Prevents multiple voting accounts in local governance
3. **Privacy Protection**: Verifies residency without storing personal documents
4. **Regulatory Compliance**: Meets legal requirements while protecting privacy
5. **Democratic Access**: Creates verifiable civic participation rights

**Technical Implementation:**

```javascript
// Privacy-preserving civic identity verification
async function verifyCivicIdentity(userId, documentType, encryptedDocument) {
  // Document verification service with zero-knowledge proofs
  const verificationService = getCivicVerificationService();
  
  // Create secure verification session
  const verificationSession = await verificationService.createSession({
    userId,
    documentType,
    regionRequired: getUserRegion(userId),
    preservePrivacy: true
  });
  
  // Upload encrypted document to verification service
  // Document never stored unencrypted
  const uploadResult = await verificationService.processDocument(
    verificationSession.id,
    encryptedDocument
  );
  
  if (uploadResult.verified) {
    // Generate zero-knowledge proof of verification
    // This proves the user was verified without revealing document details
    const zkProof = await verificationService.generateResidencyProof({
      sessionId: verificationSession.id,
      userId,
      region: uploadResult.verifiedRegion,
      expiryTime: Date.now() + (365 * 24 * 60 * 60 * 1000) // 1 year
    });
    
    // Record verification to user account (stores proof, not document)
    await userService.addRegionalVerification(userId, {
      region: uploadResult.verifiedRegion,
      verificationProof: zkProof,
      documentType: documentType,
      verifiedAt: Date.now(),
      expiresAt: zkProof.expiryTime
    });
    
    return {
      success: true,
      region: uploadResult.verifiedRegion,
      expiresAt: zkProof.expiryTime
    };
  }
  
  return {
    success: false,
    reason: uploadResult.failureReason
  };
}
```

#### **Step 12: Token Allocation and Voting Rights**

Relay's dual-token system creates a balanced approach to democratic participation and network growth. The system combines vote tokens (for governance) with invite tokens (for network growth) to create a Sybil-resistant ecosystem where democratic participation scales with community trust.

**Democratic Token Economics:**

The token system implements several key innovations:

1. **Separate Purpose Tokens**: Different tokens for different platform functions
2. **Trust-Based Access**: Governance participation requires verified identity
3. **Generational Decay**: Prevents exponential network growth that could weaken verification
4. **Engagement Incentives**: Additional tokens earned through positive community participation
5. **Progressive Access**: New users gradually gain more governance weight as trust increases

**Technical Implementation:**

```javascript
// Token allocation during onboarding
async function allocateInitialTokens(userId, inviterId) {
  // Get inviter's remaining invite tokens
  const inviterTokens = await tokenService.getInviteTokens(inviterId);
  
  // Calculate invite tokens with generational decay
  const inviteTokenAllocation = Math.max(1, inviterTokens - 1);
  
  // Standard vote token allocation for new users
  const voteTokenAllocation = 20;
  
  // Create token transactions
  const transactions = await Promise.all([
    tokenService.allocateTokens(userId, 'invite', inviteTokenAllocation, {
      reason: 'initial_allocation',
      source: inviterId,
      decayFormula: 'generational_linear'

    tokenService.allocateTokens(userId, 'vote', voteTokenAllocation, {
      reason: 'initial_allocation',
      restrictionPhase: 1,

  // Deduct invite token from inviter
  await tokenService.deductTokens(inviterId, 'invite', 1, {
    reason: 'invitation_used',
    beneficiary: userId
  });
  
  // Record token allocation to blockchain
  await blockchainService.recordTokenAllocations(transactions);
  
  return {
    inviteTokens: inviteTokenAllocation,
    voteTokens: voteTokenAllocation,
    transactions
  };
}
```
